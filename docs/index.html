整体设计

平台定位

eh2-cosim-veri 是面向 EH2（RISC-V Cores-VeeR-EH2）的 cosim-only 协同仿真验证平台。它只保留 RTL 仿真、UVM 激励、 riscv-dv、riscv compliance、覆盖率和 sign-off 所需内容，核心路径是标准 RVVI-API 在线 lockstep（逐指令比对），不是旧平台中的自定义 Cosim 多态比对层。

平台验证的主问题是：EH2 RTL 每次 retire（提交）的 PC、指令、GPR、CSR、trap 和相关内存事件，是否与 Spike 参考模型在同一 hart 上逐指令一致。

架构总图

EH2 RTL -- probe/RVFI/trace --> eh2_rvvi_adapter.sv --> rvviTrace 接口（官方）
                                                       | DPI: rvviApi.h
                                                       v
                         eh2_rvvi_scoreboard.sv -- RVVI-API 在线 lockstep
                                                       | rvviRefInit/EventStep/PcCompare/GprsCompareWritten/CsrsCompare
                                                       v
                         spike_rvvi.cc -- Spike 包成 RVVI-API 参考模型
                                                       |
                                                       v
                         spike_cosim.cc -- 复用 SpikeCosim 步进、CSR 和内存修正引擎

OPT 后 RVVI 已是默认路径。`make smoke`、`make regress TESTLIST=cosim` 和 `make signoff PROFILE=cosim` 都会默认走 RVVI lockstep；不再需要正向启用开关。

UVM agent 骨架

  • common/axi4_agent：监控和驱动 LSU、IFU、SB 三路 AXI4 总线。
  • common/trace_agent：采集 EH2 retire trace，并连接双错检测 scoreboard。
  • common/irq_agent：驱动 timer、software、external、NMI 等中断输入。
  • common/jtag_agent：驱动 JTAG debug 相关引脚。
  • common/halt_run_agent：驱动 MPC/CPU halt-run 请求。

协同仿真机制深入

单 hart lockstep 时序

  1. EH2 RTL retire 一条指令，trace/RVFI/probe 信号在 TB top 汇入 eh2_rvvi_adapter.sv。
  2. adapter 填写官方 rvviTrace 接口，包括 valid、order、PC、insn、trap、GPR 和 CSR。
  3. eh2_rvvi_scoreboard.sv 从 rvviTrace 取 retire event，并把 DUT 侧状态写入 RVVI-API。
  4. scoreboard 调用 rvviRefEventStep 让 Spike 参考模型前进一步。
  5. scoreboard 依次调用 PC、指令、写过的 GPR 和 CSR 比对函数。若 mismatch、API error 或 retire 数不一致，仿真结束时报错。

EH2 难点边界

  • 双发射 RETIRE2：adapter 按 i0/i1 提交槽生成标准 RVVI order，scoreboard 按 order 消费。
  • 异步写回：DIV、非阻塞 load 等写回可能晚于 retire 到达，scoreboard 使用 per-hart async_wb_q 按 source 和 rd 延迟匹配。
  • 压缩指令 rd 解码：SpikeCosim 中保留压缩 load 等指令的 rd 解析，用于抑制或修正部分写回场景。
  • CSR 豁免和修正：标准 CSR 比对集中在 mstatus、mtvec、mepc、mcause、mtval、mip 等路径；EH2 自定义 CSR 在 spike_cosim.cc 中预注册和 WARL 修正。不要把这理解成完整 SoC CSR 建模。
  • store 通知：DUT store 总线事件通过 rvviDutBusWrite 传给参考侧，用于对齐 Spike 的内存视图。

双 hart SMT

双 hart 使用 CONFIG=dual_thread 或 CONFIG=default_mt 切到 rtl/snapshots/default_mt，Makefile 同时设置 RVVI_NHART=2。 运行时仍需要传 +rvvi_nhart=2，scoreboard 会检查 plusarg 与编译出的 NHART 是否一致。 

make smoke CONFIG=dual_thread SIM_OPTS="+rvvi_nhart=2"

当前范围是两个 hart 各自的核内 lockstep：per-hart 队列、net、bus 和 retire 路径分别比对。核间 DCCM 窗口一致性比对不在本平台范围内。

RVVI-API 接口映射

scoreboard 与 spike_rvvi.cc 之间通过官方 RVVI-API（DPI 函数，声明在 vendor/rvvi/include/host/rvvi/rvviApi.h）通信。函数分两侧：DUT 侧把 RTL 观测到的状态推给 RVVI；Ref 侧驱动、查询、比对 Spike 参考模型。下表是 lockstep 主循环用到的核心函数（完整 API 见上游头文件）。

DUT 侧（RTL → RVVI）

函数作用
rvviDutRetire提交一条 DUT retire（hart, order, PC, insn）。
rvviDutTrap提交一次 DUT trap 事件。
rvviDutGprSet登记 DUT 本条 retire 写回的 GPR 值。
rvviDutCsrSet登记 DUT 侧 CSR 值。
rvviDutBusWrite把 DUT store 总线写（地址/数据/byte-enable）通知参考侧，对齐内存视图。
rvviDutCycleCountSet登记周期计数。

Ref 侧（驱动 / 比对 Spike）

函数作用
rvviRefInit初始化参考模型、加载 ELF；rvviRefConfigSetInt 配 hart 数等。
rvviRefPcSet / rvviRefGprSet / rvviRefCsrSet预置参考侧 PC / GPR / CSR 初值（含 28 自定义 CSR 预注册）。
rvviRefEventStep让 Spike 步进一条指令。
rvviRefEventComplete标记本次 retire 事件处理完成。
rvviRefPcCompare比对 PC。
rvviRefInsBinCompare比对指令编码。
rvviRefGprsCompareWritten只比对本条写过的 GPR（配合异步写回延迟匹配）。
rvviRefCsrsCompare / rvviRefCsrCompare比对 CSR。
rvviRefCsrCompareEnable / rvviRefCsrCompareMaskCSR 豁免窗开关 / 掩码（自定义 CSR 不参与比对）。
rvviRefNetSet异步注入中断 / NMI / debug 请求 net。
rvviRefMemoryRead / rvviRefMemoryWrite参考侧内存读写。

EH2 自定义 CSR 豁免清单

EH2 含一批非标准 CSR，Spike 默认会对它们抛 illegal-instruction trap。平台在 dv/cosim/spike_cosim.cc 中预注册这些 CSR（唯一源），使 Spike 能读写但不纳入 lockstep 比对（经 rvviRefCsrCompareEnable 豁免）。共约 29 个，按功能分组：

分组CSR（地址 名称）
电源 / 缓存 / 特性控制0x7C0 mrac、0x7F9 mfdc、0x7F8 mcgc、0x7C6 mpmc、0x7C2 mcpc、0x7FF mscause
SMT / hart 控制0x7FC mhartstart、0x7FE mnmipdel、0xFC4 mhartnum
内部定时器0x7D2 mitcnt0、0x7D5 mitcnt1、0x7D3 mitb0、0x7D6 mitb1、0x7D4 mitctl0、0x7D7 mitctl1
快速外部中断 / PIC0xBC8 meivt、0xFC8 meihap、0xBC9 meipt、0xBCA meicpct、0xBCC meicurpl、0xBCB meicidpl
内存 ECC 错误0x7F0 micect、0x7F1 miccmect、0x7F2 mdccmect、0xBC0 mdeau、0xFC0 mdseac
Debug 辅助0x7C4 dmst、0x7CE mfdht、0x7CF mfdhs
标准 trap CSR（mstatus/mtvec/mepc/mcause/mtval/mip）参与比对。 其余未在 adapter surface 暴露的标准 CSR 当前未比对——这是覆盖边界，不是 bug，扩展全 CSR 比对是后续工作。

核配置 Profile

eh2_configs.yaml 定义多套 EH2 DUT 配置，由 CONFIG 选择。 单/双线程影响 RVVI_NHART 与 snapshot 选择。

CONFIG线程说明
default1默认：AXI4、单线程、全特性（DCCM/ICCM 64KB、ICache 32KB、Atomic、Bitmanip Zba/Zbb/Zbc/Zbs、PIC 127 中断、BHT/BTB 512）。
dual_thread / default_mt2双线程 SMT：触发 RVVI_NHART=2 + snapshot rtl/snapshots/default_mt，运行时配 +rvvi_nhart=2。
minimal1精简：无 ICache/DCCM/ICCM、无 Atomic/Bitmanip、PIC 16 中断。
ahb_lite1AHB-Lite 总线（非 AXI4）配置。
cosim lockstep 主路径在 default 与 dual_thread 上验证；minimal/ahb_lite 为配置维度保留。

sign-off 设计

入口是 dv/uvm/core_eh2/scripts/signoff.py，允许的 stage 固定为： smoke、directed、cosim、riscvdv、 compliance。覆盖率 gate 由 Makefile 的 COV 和 signoff 参数控制。

Profile Stage 集合
quicksmoke、directed
cosimsmoke、cosim
riscvdv_smokeriscvdv
nightlysmoke、directed、cosim、riscvdv
fullsmoke、directed、cosim、riscvdv、compliance

当前最小通过数阈值来自 STAGE_MIN_PASSED：smoke=1、 directed=33、cosim=7、riscvdv=50、 compliance=85。

功能覆盖率体系

除仿真器自带的 line/toggle/FSM/branch/assert 覆盖率（由 COV=1 开启）外，平台在 dv/uvm/core_eh2/fcov/ 下维护功能覆盖率：

文件职责
fcov/eh2_fcov_if.sv核心功能覆盖 interface（指令类型、特权模式、trap、异步写回等 covergroup）。
fcov/eh2_pmp_fcov_if.svPMP 专项功能覆盖（TOR/NAPOT/NA4/OFF 模式、R/W/X 组合、跨区、锁定等，约 1461 行）。
fcov/eh2_fcov_bind.sv把 fcov interface bind 到 RTL 层级。
fcov/eh2_csr_categories.svhCSR 分类定义，供覆盖率/比对引用。
fcov/cov_waivers/覆盖率豁免：eh2_cov_waiver_pkg.sv + 不可达 cross 豁免 yaml（dual-issue presync stall、NMI-during-debug 等）。

VCS/NC 覆盖率配置文件在 TB 根：cover.cfg（层级）、cov_fsm.cfg / cov_fsm_reset_filter.cfg（FSM）、cov_full_nc.ccf（NC 覆盖配置）。 签核覆盖率门限由 SIGNOFF_MIN_LINE_COV / SIGNOFF_MIN_FUNCTIONAL_COV 控制。

自包含与 vendoring

项目级依赖全部在 vendor/ 下：vendor/spike、 vendor/google_riscv-dv、vendor/rvvi、 vendor/riscv-tests。make spike 会在库内构建 Spike，并生成 vendor/spike/install；该产物不进 Git。

允许外部提供的机器前提只有仿真器（VCS 或 Incisive/irun）、支持 C++17 的 g++、 以及 riscv32-unknown-elf-* 工具链。机器路径写在本地 env.mk， env.mk 被 Git 忽略。

使用方式

机器前提

  • Synopsys VCS，或 Cadence Incisive（irun）。
  • 支持 C++17 的 g++。系统 g++ 4.8.5 不满足 Spike/RVVI 侧构建要求。
  • riscv32-unknown-elf-* 工具链，或通过 RISCV_PREFIX 指定前缀。
提示：VCS / Incisive、C++17 g++、riscv32 工具链的具体路径与版本因机器而异，请按本机实际安装情况在 env.mk 中配置。

首次配置

  1. 复制本机配置模板：cp env.mk.example env.mk。
  2. 填写 SPIKE_CXX，它必须指向支持 C++17 的 g++。
  3. 填写 NC_INSTALL，用于找到 Incisive 的 svdpi.h 和 UVM-1.2；如果只用 VCS，也可以设置 VCS_HOME 作为 svdpi.h fallback。
  4. 如工具链命令不是默认 riscv32-unknown-elf- 前缀，设置 RISCV_PREFIX。
  5. 运行 make spike，在 vendor/spike 内构建 Spike install。
cp env.mk.example env.mk
# 编辑 env.mk：SPIKE_CXX、NC_INSTALL；必要时设置 VCS_HOME、RISCV_PREFIX
make spike

SPIKE_DIR、SPIKE_INSTALL、RISCV_TESTS_FW、 RISCV_COMPLIANCE_FW 在 Makefile 中默认指向库内 vendored 目录，通常不需要写进 env.mk，除非你明确要覆盖。

make spike 首次在库内构建 Spike，耗时取决于机器，通常为分钟级；构建产物 vendor/spike/install 会被复用，后续无需重建。

快速上手

最小闭环是 make smoke。它会构建 tests/asm/smoke.hex / smoke.elf、编译 TB、启动仿真，并默认启用 RVVI lockstep。 

make smoke

通过条件是回归结果 Passed: 1 | Failed: 0，并且仿真日志中 RVVI_SCOREBOARD 结束统计的 mismatches=0。

make help

打印当前 Makefile 的精简入口说明。

前置依赖：无。接受变量：无。

make help
make -n smoke
make -n signoff PROFILE=cosim

make spike

在 vendor/spike 内配置、编译并安装 Spike，产出 vendor/spike/install。

前置依赖：无。接受变量：SPIKE_DIR、SPIKE_INSTALL、SPIKE_CXX、SPIKE_CXXFLAGS、SPIKE_BUILD。

make spike
make spike SPIKE_CXX=/path/to/cxx17/g++
make spike SPIKE_DIR=$PWD/vendor/spike

make cosim

编译 RVVI-only 协同仿真的 DPI 共享库 build/libcosim.so，源文件是 dv/cosim/spike_cosim.cc 和 dv/cosim/spike_rvvi.cc。

前置依赖：$(LIBCOSIM) 文件目标；该文件目标要求 SPIKE_INSTALL 已存在。接受变量：SPIKE_DIR、SPIKE_INSTALL、SPIKE_CXX、SPIKE_CXXFLAGS、SPIKE_BUILD、NC_INSTALL、NC_UVM_HOME、SVDPI_INCLUDE。

make cosim
make cosim NC_INSTALL=/path/to/INCISIVE
make cosim SVDPI_INCLUDE=/path/to/svdpi.h

make rvviref

构建并运行独立 Spike RVVI 参考模型，不启动 RTL 仿真，适合快速看参考侧 retire trace。

前置依赖：asm $(RVVIREF_EXE)。接受变量：BINARY、RVVIREF_STEPS、RVVIREF_LOG、SPIKE_DIR、SPIKE_INSTALL、SPIKE_CXX、SPIKE_CXXFLAGS、SPIKE_BUILD。

make rvviref
make rvviref BINARY=tests/asm/smoke.elf RVVIREF_STEPS=20
make rvviref RVVIREF_LOG=build/rvviref/custom.log

make asm

进入 tests/asm 编译汇编测试，生成 .hex、.elf 和 .dis。

前置依赖：无。接受变量：RISCV_PREFIX。

make asm
make asm RISCV_PREFIX=riscv32-unknown-elf-

make compile

编译 RTL、共享 AXI4 RTL 和 UVM TB。SIMULATOR=vcs 时分发到 compile_vcs，SIMULATOR=nc 时分发到 compile_nc。

前置依赖：compile_$(SIMULATOR)；compile_vcs / compile_nc 的显式依赖是 $(LIBCOSIM) | $(BUILD_DIR)。接受变量：SIMULATOR、CONFIG、BUILD_SUBDIR、COV、WAVES、NC_INSTALL、NC_UVM_HOME、SVDPI_INCLUDE。

make compile
make compile SIMULATOR=nc
make compile CONFIG=dual_thread BUILD_SUBDIR=build/compile_dual

make smoke

构建汇编、编译 TB，然后运行 tests/asm/smoke.hex。这是最小 RVVI lockstep 闭环。

前置依赖：asm。接受变量：SIMULATOR、CONFIG、SIM_OPTS、COV、WAVES、TIMEOUT_NS、VERBOSITY。

make smoke
make smoke SIMULATOR=nc
make smoke CONFIG=dual_thread SIM_OPTS="+rvvi_nhart=2"

make regress

按 testlist 调用 run_regress.py。TESTLIST=cosim 对应 dv/uvm/core_eh2/directed_tests/cosim_testlist.yaml。

前置依赖：目标行无显式依赖，命令内部先调用 compile。接受变量：TESTLIST、TEST、SEED、ITERATIONS、PARALLEL、SIMULATOR、SIM_OPTS、OUT、COV、WAVES、TIMEOUT_NS、VERBOSITY。

make regress TESTLIST=cosim
make regress TESTLIST=cosim TEST=cosim_alu
make regress TESTLIST=directed PARALLEL=8

make compliance

调用 dv/uvm/riscv_compliance 下的 compliance 集成。默认 suite 语义是 run。

前置依赖：目标行无显式依赖；当 SUITE 不是 compile 时，内部先调用 compile。接受变量：SUITE、RISCV_COMPLIANCE_FW、RISCV_TESTS_FW、RISCV_PREFIX、SIMULATOR、COMPLIANCE_BUILD_SUBDIR、COMPLIANCE_SIMV、NC_INSTALL。

make compliance
make compliance SUITE=compile
make compliance SUITE=all

make watch_wave

运行指定汇编测试并打开波形。MODE=batch 语义是跑完离线查看；MODE=live 使用 NC GUI 边仿真边看，需要图形环境。

前置依赖：batch 分支为 asm，live 分支为 asm cosim。接受变量：TEST、MODE、SIMULATOR、CONFIG、SIM_OPTS、TIMEOUT_NS、WAVES、NC_INSTALL、NC_UVM_HOME。

make watch_wave TEST=smoke
make watch_wave TEST=smoke SIMULATOR=nc
make watch_wave TEST=smoke MODE=live

make signoff

运行 cosim-only 签核入口 dv/uvm/core_eh2/scripts/signoff.py，stage 由 PROFILE 决定。

前置依赖：目标行无显式依赖；非 GATE_ONLY=1 时内部调用 asm 和 compile。接受变量：PROFILE、SIMULATOR、GATE_ONLY、COV、PARALLEL、SEED、SIGNOFF_OUT、SIGNOFF_OPTS、SIGNOFF_ITERATIONS、SIGNOFF_MIN_LINE_COV、SIGNOFF_MIN_FUNCTIONAL_COV、SIGNOFF_ALLOW_WARNINGS、WAVES、CLEANUP。

典型工作流

# 调试单个 cosim 测试
make regress TESTLIST=cosim TEST=cosim_alu

# 改用 NC 并打开波形
make smoke SIMULATOR=nc WAVES=1

# 双 hart 冒烟
make smoke CONFIG=dual_thread SIM_OPTS="+rvvi_nhart=2"

# 快速签核：smoke + cosim
make signoff PROFILE=cosim

# 全量签核：默认 full
make signoff

# 离线看波形，VCS 分支会打开 Verdi
make watch_wave TEST=smoke

# NC 实时看波形，需图形环境
make watch_wave TEST=smoke MODE=live

# compliance 只编译
make compliance SUITE=compile

# 清理、彻底清理、预览清理
make clean
make clean FORCE=1
make clean DRY_RUN=1
耗时参考：make smoke 通常 1–2 分钟；make signoff PROFILE=cosim 数分钟；make signoff（默认 full，含 riscvdv 与 compliance）较长，请预留充足时间。具体随机器与仿真器而异。

测试体系

平台测试分四层：cosim 证明测试（RVVI lockstep 框架证明点）、directed 定向测试 （汇编激励，覆盖具体 RTL 特性）、UVM 测试类（约 56 个 core_eh2_*_test）、 riscv-dv 随机（约束随机指令流）。

cosim 证明测试（TESTLIST=cosim，7 个）

均以 core_eh2_rvvi_test 运行，构造上即开启 Spike lockstep；签核要求全部通过（STAGE_MIN_PASSED[cosim]=7）。

测试覆盖点
cosim_smoke初始化、加载二进制、首次 Spike step、mailbox PASS。
cosim_alu确定性 ALU 指令的寄存器写回相关性。
cosim_load_storeLSU AXI 内存通知路径（确定性 load/store）。
cosim_dual_issueEH2 双发射 retire trace 的程序序 lockstep。
cosim_bitmanipZba/Zbb 指令 lockstep（sh1add/andn/clz/max 等）。

新增一个 cosim 定向测试

  1. 在 tests/asm/ 新增一个 .S 文件。
  2. 在 dv/uvm/core_eh2/directed_tests/cosim_testlist.yaml 注册测试条目，复用 eh2_cosim config。
  3. 运行 make regress TESTLIST=cosim TEST=<name>。

现有 cosim_alu 条目可作为最小示例：

- test: cosim_alu
  desc: "Register writeback correlation for deterministic ALU instructions"
  config: eh2_cosim
  test_srcs: tests/asm/cosim_alu.S
  iterations: 1

失配排查

  • 先看仿真日志中的 RVVI_SCOREBOARD 报错。PC、insn、GPR、CSR mismatch 会带出 hart、retire slot、PC 和指令值。
  • 用对应 .dis 反汇编定位 retire 指令，再回看 rvvi_trace.log 和波形。
  • 如果是 CSR 差异，先确认是否属于 EH2 自定义 CSR 或 WARL 字段；当前 CSR 预注册和修正集中在 dv/cosim/spike_cosim.cc。
  • 如果是异步写回，检查 DIV、非阻塞 load 或 suppressed write 相关路径，重点看 async_wb_q 入队和匹配条件。
